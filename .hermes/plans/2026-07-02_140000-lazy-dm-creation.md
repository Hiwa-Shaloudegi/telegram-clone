# Lazy DM Creation + Send/Delete Messages in Private Chats

> **For Hermes:** Use subagent-driven-development skill to implement this plan task-by-task.

**Goal:** Navigate from contacts to a "pending" DM page without creating a database row; create the DM only when the first message is sent. Also verify and fix sending and deleting messages in private chats.

**Architecture:** When a contact is tapped, navigate to a `PendingDmPage` (route `/dm/:otherUserId`) using a `PendingDmExtra` with the contact's display info. The page shows the chat UI (app bar, empty message list, input bar) backed by a synthetic `ChatListItemModel` тАФ no database calls. On first send, a `SendFirstDmMessageCommand` calls `get_or_create_private_chat` then `send_message`, returns the real `chatId`, and the page replaces itself with `/chat/:chatId`. All subsequent messages and deletes use the existing `send_message` / `bulk_delete_messages` RPCs.

**Tech Stack:** Flutter, Riverpod (code-gen), GoRouter, Supabase (RPC + realtime), build_runner

## Current State Analysis

### What exists
- `CreatePrivateChatCommand` (`lib/features/chat_list/notifiers/command/create_private_chat_command.dart`) тАФ calls `get_or_create_private_chat` RPC then navigates to `/chat/:chatId`. This creates the DM in the DB immediately on contact tap.
- `contacts_list.dart` line 45-50 тАФ calls `createPrivateChatCommandProvider.run(otherUserId, router)` on contact tap.
- `ChatPage` (`lib/features/chat/ui/pages/chat_page.dart`) тАФ watches messages via `watchMessagesQueryProvider(chatId)`, handles send/edit/delete via command notifiers.
- `SendMessageCommand` тАФ optimistic update + `send_message` RPC. Works for existing chats.
- `DeleteMessagesCommand` тАФ optimistic removal + `bulk_delete_messages` RPC. Works via selection mode in `SelectionAppBar`.
- `MessagesApi.deleteMessage` тАФ single-message soft delete (sets `deleted_at`).
- `MessageBubble._showContextMenu` тАФ dead code, never called. Long-press enters selection mode instead.

### What's broken / missing
1. **"No chat selected"**: When `CreatePrivateChatCommand` creates the DM and navigates, `ChatPage` can't find the chat in `watchUserChatsQueryProvider` stream (not refreshed yet) and `mainUi_selectedChatItemProviderProvider` is null (contacts_list never sets it). Falls back to a bare `ChatListItemModel` with `otherUserName: null` тЖТ title shows "Unknown".
2. **Lazy creation**: User wants the DM row created only on first message, not on contact tap.
3. **Context menu delete**: `_showContextMenu` in `message_bubble.dart` is dead code тАФ the Delete action does nothing.
4. **Sending/deleting verification**: Code exists but needs end-to-end verification for private chats.

## Task 3: Create `SendFirstDmMessageCommand` notifier

**Objective:** Riverpod command that calls `get_or_create_private_chat` then `send_message`, returning the real `chatId` so the caller can navigate.

**Files:**
- Create: `lib/features/chat/notifiers/command/send_first_dm_message_command.dart`

**Step 1: Create the file**

```dart
import 'dart:async';

import 'package:riverpod_annotation/riverpod_annotation.dart';
import 'package:telegram_clone/data/api/chat/chats_api.dart';
import 'package:telegram_clone/data/api/messages/messages_api.dart';

part 'send_first_dm_message_command.g.dart';

@riverpod
class SendFirstDmMessageCommand extends _$SendFirstDmMessageCommand {
  @override
  FutureOr<void> build() {}

  Future<String?> run({
    required String otherUserId,
    required String content,
    String? replyToMessageId,
  }) async {
    final link = ref.keepAlive();
    state = const AsyncValue.loading();

    String? chatId;

    state = await AsyncValue.guard(() async {
      chatId = await ref
          .read(chatsApiProvider)
          .getOrCreatePrivateChat(otherUserId);

      await ref.read(messagesApiProvider).sendTextMessage(
            chatId: chatId!,
            content: content,
            replyToMessageId: replyToMessageId,
          );
    });

    link.close();
    return chatId;
  }
}
```

      context.pushReplacementNamed(
        RouteNames.chat,
        pathParameters: {'chatId': chatId},

                      fontSize: 16,

## Risks, Tradeoffs, and Open Questions

### Open Question 1: Long-press gesture conflict

Currently long-press enters selection mode. Task 7 changes long-press to show the context menu. This means users can no longer enter selection mode via long-press.

**Options:**
- A) Add a "Select" option to the context menu that enters selection mode
- B) Keep long-press for selection mode, add a different gesture (e.g., right-click or a menu button) for the context menu
- C) Long-press shows context menu; context menu has "Select" option that enters selection mode and auto-selects the message

**Recommendation:** Option C is most Telegram-like. Add a "Select" ListTile to `_showContextMenu` that calls `ref.read(chatUi_selectedMessagesProvider.notifier).toggle(message.id)`.

### Open Question 2: `get_or_create_private_chat` idempotency

If the user navigates to a pending DM, sends a message (which creates the chat), goes back, and taps the same contact again тАФ the `PendingDmPage` will show again (empty messages). But when they send, `get_or_create_private_chat` will return the existing chat, and the message will be added to it. The user won't see previous messages until they navigate to the real chat. This is acceptable but could be improved by checking for an existing DM before showing the pending page.

### Risk: Race condition after first message

After `SendFirstDmMessageCommand` creates the chat and sends the message, the `watchUserChatsQueryProvider` stream may not have the new chat yet. We mitigate this by setting `mainUi_selectedChatItemProviderProvider` before navigating. The `ChatPage` uses this provider first, so it should have the chat info.

### Risk: `SendFirstDmMessageCommand` failure

If `get_or_create_private_chat` succeeds but `send_message` fails, the DM is created but has no messages. The user stays on `PendingDmPage` with `_isSending = false`. On retry, `get_or_create_private_chat` returns the existing chat (idempotent), and `send_message` is called again. This is acceptable тАФ no orphaned chats since the chat will eventually appear in the chat list.

### Tradeoff: No optimistic message in PendingDmPage

The `PendingDmPage` doesn't show an optimistic message because there's no `watchMessagesQueryProvider` (no real chatId). The user sees a loading spinner briefly, then the real chat page. This is a minor UX tradeoff for simplicity.

### Not modified: `CreatePrivateChatCommand`

The `CreatePrivateChatCommand` and `create_private_chat_command.dart` are left in place but no longer called from `contacts_list.dart`. They can be removed in a future cleanup if desired.
